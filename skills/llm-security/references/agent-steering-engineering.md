# Agent 引导工程 — 让工作流指令可靠落地

> 来源：2026 年多来源综合（Anthropic Skill Engineering、Microsoft Code Words、Strands Steering Hooks、Gradient Flow Harness Engineering）
> 适用场景：AI 编码 Agent（Claude Code / Codex / Cursor / Cline / Windsurf / Kiro 等）读完 README / RULES 后只确认不执行、跳过步骤、自作主张省略关键操作。
> 定位：这是**指令可靠性工程**——让写下的流程被稳定执行的一组设计模式，也是本技能包自身的设计依据。

---

## 核心问题诊断

Agent"读了工作流但不照做"的根因不是模型能力，而是自然语言指令存在语义逃逸空间：

| 根因 | 解释 |
|------|------|
| 上下文注意力衰减 | 长文档中部内容被注意力机制降权，Agent 实际只"看到"开头和结尾 |
| 语义覆盖 | 模型优化"帮助性"时会创造性重解释显式指令（把 MUST DO X 理解成"建议做 X"） |
| 被动语言被当可选 | "Ready for next step → invoke X" 被当作建议而非指令 |
| 无状态强制 | 缺少外部状态机验证工作流顺序，跳步不会被发现 |
| 沉默状态腐化 | 产出结构正确但语义错误的结果，错误静默累积 |

## 技术 1：指令置顶（Critical-First Pattern）

把"下一步做什么"放在最前面，上下文放在后面：

```text
效果差（指令被忽略）:
  [70 行项目背景和工具列表]
  → "下一步：运行 bootstrap 安装缺失工具"

效果好（指令被执行）:
  "## 启动序列：运行 bootstrap-reverse 检查并安装缺失工具
   → 完成后读取 routing.md 确定入口技能"
  [然后是项目背景和工具列表]
```

原理：LLM 对提示词首尾内容赋予最高注意力权重，中间内容可能被完全忽略。

本项目应用：RULES.md 的"启动序列"置于全文开头；每个 SKILL.md 的第一个章节是"操作协议"而非"适用场景"。

## 技术 2：指令性语言（Directive over Suggestive）

将建议性语言替换为 RFC 2119 级别的指令性语言：

| 弱语言（易被跳过） | 强语言（语义明确） |
|---|---|
| "可以尝试..." | MUST: 执行... |
| "Ready for next step → invoke X" | REQUIRED: 调用 X，无需等待确认 |
| "建议先读 routing.md" | REQUIRED: 进入子模块前先读完 routing.md |
| "如果缺工具可以 bootstrap" | 缺工具时的标准动作是调用 bootstrap |
| "记得更新 field-journal" | 任务完成后逐项执行完成清单 |

关键约定：

```text
MUST / MUST NOT — 强制性约束，违背即失败或越界
SHOULD          — 默认应当，不执行需说明理由
MAY             — 真正可选
```

注意：强关键词是稀缺资源。全文都是 MUST 等于没有 MUST——只为真正影响成败的约束保留。

## 技术 3：常见失效模式与对策

Agent 在长任务中有若干可预测的失效模式。设计指令时预先针对：

| 失效模式 | 对策 |
|---|---|
| 静默跳步（"这一步可以省略"） | 要求跳步必须显式说明理由并告知用户 |
| 自作主张（"根据我的判断不需要"） | 要求列出判断依据；明确写出的步骤不可凭判断跳过 |
| 替用户做决定（"用户大概不需要"） | 决策点呈现全部选项，可标推荐但不隐藏备选 |
| 凭记忆行动（"我知道怎么做，不用读 X"） | 先读 X 再行动——文件可能携带任务特定约束 |
| 以跳步换速度 | 引导为并行执行独立步骤，而非跳过 |
| 凭经验猜路径 | 路径一律来自 tool-index；不同机器安装位置不同 |
| 跳过收尾清单 | 任务完成的定义 = 清单逐项完成 |
| 缺索引时猜路径 | tool-index 缺失时先运行 refresh 脚本生成 |
| 默认省掉报告 | 报告是默认产物，除非用户明确说不要 |
| 简单任务不记录 | 至少一行：目标类型 + 用了什么 + 有无意外 |

使用方式：将本表放在指令文件的高注意力区域（开头或结尾 10%）。

## 技术 4：Skill 工程五模式（Anthropic 2026）

| 模式 | 适用场景 | 关键技巧 |
|---|---|---|
| Linear Flow 线性流 | 步骤清晰的流程（部署、安装） | 提供安全默认值，显式否定指令（"MUST NOT use --force"） |
| Decision Tree 决策树 | 平台导航、故障诊断 | 树形导航 + `references/` 渐进加载 |
| Iterative Loop 迭代循环 | TDD、审查-修复循环 | 硬规则前置 + 失效模式对策阻断走捷径 |
| Baton Loop 接力循环 | 多会话、多 Agent 协作 | 状态外化到 journal / `next-prompt.md` |
| Multi-Phase + Checkpoints | 多天复杂工作流 | 编排器"父"skill + 人工 Go/No-Go 检查点 |

本项目映射：

- 标准行为链 = Linear Flow（按序执行）
- 路由矩阵 = Decision Tree（三轴匹配）
- 完成清单 = Multi-Phase Checkpoint
- Field Journal = Baton Loop（跨会话状态外化）

## 技术 5：In-Band 自校验（Steering Hooks 思想）

不依赖 Agent"自觉"，而是在指令中嵌入自我校验：

```text
每次声称"任务完成"前，先自审：
1. 行为链中是否有被跳过的步骤？哪一步？
2. 是否有凭记忆猜的工具路径？实际的 tool-index 路径是什么？
3. 完成清单是否逐项完成？未完成的项原因是什么？
4. 以上任何一项存疑 → 任务未完成，回到对应步骤。
```

自审计比外部校验更即时——它发生在"说出口"之前。

## 技术 6：不透明标识符（Code Words）

Microsoft 2026 研究：语义化参数名会触发模型"帮忙优化"的倾向：

```text
WRONG:   { "query": "...", "top": 9 }        → 68.4% 参数遵循率
CORRECT: { "query": "...", "code": "alpha" } → 100% 参数遵循率
```

应用：bootstrap 精确配置、工具调用中需要强保证的参数，使用 code word 映射表。

## 技术 7：双 AI 审查回路（Dual Validation）

```text
AI A（执行者）产出结果
  ↓
AI B（审查者）对照规则检查
  ↓ 通过                ↓ 不通过
输出给用户          退回修正，附具体违规引用
```

本项目应用：输出报告前，先对照完成清单逐项自查；发现缺项回到对应步骤补做。

## 技术 8：上下文窗口布局优化

LLM 注意力分布（高→低）：

```text
[开头 10%] ████████████ ← 最高注意力：放立即行动指令
[中部 80%] ████░░░░░░░░ ← 注意力递减：放参考材料
[结尾 10%] ████████████ ← 注意力回升：放清单与不可跳过项
```

具体应用：

1. 开头 10%：启动序列 + 触发关键词
2. 中部 80%：详细工作流、参考链接、工具列表
3. 结尾 10%：失效模式对策 + 完成清单 + 反模式清单

## 实践模板

### 模板 A：技能入口的操作协议

```markdown
## 操作协议

1. 确认当前任务命中本技能的适用范围。
2. 阅读 `../tool-index.md`，校验工具可用性与实际路径。
3. 缺工具时调用 bootstrap，不猜路径。
4. 进入"工作流"第一步并执行。
5. 不确定时列出具体问题，不沉默。
```

### 模板 B：任务完成自检

```markdown
## 完成前自检

- [ ] 行为链的每一步都已执行（而非只阅读）
- [ ] 工具路径全部来自 tool-index.md
- [ ] 产出了可复现的命令 / 脚本 / 报告
- [ ] 经验已回写 field-journal（如有踩坑）
- [ ] 完成清单（报告 + 图表 + 经验回写）已执行
```

### 模板 C：反模式清单

```markdown
## 反模式

- 读完规则只回复"已理解，请告诉我任务" → 应主动路由并开始
- 声称"步骤 1–4 已完成"但实际只是读过 → 区分"已读"与"已执行"
- 未执行完成清单就宣布完成 → 清单是完成的定义
- 用"根据经验"替代查 tool-index → 路径因机器而异
```
